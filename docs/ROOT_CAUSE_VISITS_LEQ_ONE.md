# 근본 원인: 자식 노드 방문이 1 이하로만 쌓이는 이유

방문 상한이 “작아서”가 아니라, **루트 총 방문 수가 적게 쌓이기 때문**이다. 그게 적은 이유만 따로 정리한다.

---

## 1. 왜 “방문 1 이하”가 나오나

- `extractAnalysisData` / `getAnalysisData`가 주는 건 **루트의 자식들(각 수)** 이고, 각 항목의 `numVisits`는 **그 수로 나간 플레이아웃 수(edge visits)** 이다.
- 따라서 **모든 자식의 numVisits 합 = 루트 총 방문 수** 이다.
- 빈 바둑판이면 합법 수가 361개라, 루트 총 방문이 361보다 작으면 **평균 1 미만** → 대부분 자식이 0 또는 1만 가질 수밖에 없다.
- 그래서 **“자식 방문이 1 이하”** 인 현상은, **루트 총 방문 수가 (합법 수 대비) 너무 적게만 쌓였을 때** 생긴다.

즉, “상한이 작다”가 직접 원인이 아니라 **“실제로 쌓인 루트 방문 수가 적다”** 가 직접 원인이다.

---

## 2. 루트 방문이 적게 쌓이는 이유 (근본 원인 후보)

### (1) 방문 상한으로 일찍 끊김 — 제거함

- 앱에서 `maxVisits`를 128 / 1000 / 5000 등으로 넘기면, 네이티브에서 **그 수에 도달하는 순간** 검색이 멈춘다.
- 루트 총 방문이 128이면, 361개 수에 나눠지므로 자식당 1 이하는 당연한 결과다.
- **조치**: 방문 제한을 없애고 **시간(budgetMs)만으로 제어**하도록 변경했다. `maxVisits = 0` 이면 네이티브에서 time-only로 동작해, 주어진 시간만큼만 돌리므로 루트 방문이 더 쌓인다.

### (2) 검색이 중간에 멈춤 (g_stopFlag)

- 네이티브는 `runWholeSearch(g_stopFlag)` 로 돌고, **`g_stopFlag`가 true가 되면** 시간이 남아 있어도 검색을 중단한다.
- `stopAnalysis()`가 호출되면 `g_stopFlag`가 true가 되고, 그 시점까지 쌓인 루트 방문만 남는다.
- **호출 위치**:
  - **세션 전환**(분석 ↔ 대국 탭): `EngineOrchestrator`에서 `SwitchSession` 이벤트 처리 시 `stopAnalysis(ANALYSIS)`, `stopAnalysis(PLAY)` 호출.
  - **앱 생명주기**: `KataGoApp`에서 trim memory / task removed 시 `stopAnalysis` 호출.
  - **ViewModel 정리**: `onCleared`에서 두 세션 모두 `stopAnalysis` 호출.
- 따라서 **탭을 자주 바꾸거나**, **백그라운드/메모리 정리 등으로 stop이 자주 걸리면**, 시간 예산을 다 쓰기 전에 검색이 끊겨 루트 방문이 적게 쌓일 수 있다.

### (3) 같은 프로세스에서 다음 분석이 올 때

- 한 번에 하나의 `analyzeTop4`만 돌고, 끝나야 다음이 시작된다.  
  중간에 **다른 경로로** `stopAnalysis`가 호출되면, 그때 돌고 있던 검색만 조기 종료된다.
- “의도만 자주 바뀌고 실행은 한 번”인 구조라면, **실행 중인 검색을 멈추는 일은 (1)(2) 경로에서만** 발생한다.

---

## 3. 정리

| 현상 | 직접 원인 | 근본 원인 (루트 방문이 적은 이유) |
|------|-----------|-----------------------------------|
| 자식 노드 numVisits가 전부 ≤1 | 루트 총 방문 수 < 합법 수(예: 361) | (1) **방문 상한**으로 검색이 일찍 끊김 → **해결: 시간만 제어하도록 변경** (2) **stopAnalysis**로 검색이 중간에 끊김 (탭 전환, 메모리/생명주기 등) |

방문 상한은 제거했으므로, 이후에도 “방문 1 이하”가 반복되면 **(2) 검색이 중간에 멈추는 경우**를 의심하면 된다 (탭 전환 직후, 백그라운드 전환 직후 등).
