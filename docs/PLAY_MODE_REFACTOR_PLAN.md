# 대국 모드 근본 안정화 — 리팩터링 계획

## 핵심 목표

- 대국 모드를 **“타이밍 레이스가 없는 단일 상태 기계”**로 재구성한다.
- UI / 엔진 / 정책 / Undo가 각자 상태를 움직여 충돌하는 현재 구조를 제거한다.

---

## 근본 해결 방향 (설계 수준)

### 1. 단일 진실 (Source of Truth)

- `moves`와 `stones`를 동시에 보관하면서 각각 다른 시점에 갱신하는 구조를 없앤다.
- **하나만 진실**로 두고 나머지는 **파생 계산**으로 만든다.
  - 예: `moves`만 저장 → `stones`는 필요할 때 재계산(캐시)
  - 또는 `stones`만 저장 → `moves`는 로그로만 사용
- 현재 불안정성은 “moves 갱신 vs stones 갱신” 타이밍 불일치에서 온다.

### 2. 원자적 착수 파이프라인 (Atomic Move Pipeline)

- 착수는 **하나의 트랜잭션**으로 처리한다.  
  **입력 → 합법성 체크 → 확정 → UI 반영**
- 낙관 업데이트(먼저 UI에 수를 놓는 것)를 제거하거나,  
  **확정된 결과만 UI에 반영**하도록 바꾼다.
- “먼저 보여주고 나중에 되돌림”이 모든 레이스를 유발한다.

### 3. 엔진 의존 완화

- 엔진이 준비되지 않은 상황에서도 **합법성 검사**는 반드시 일관되게 수행된다.
- 엔진 준비 여부에 따라 합법성 검사가 달라지는 구조는 근본적으로 불안정하다.
- 최소한 **로컬 룰(SimpleGoBoard)**로 항상 동일한 합법성 판단을 제공해야 한다.

### 4. 상태 머신 도입

- 대국 모드 상태를 **명확히 고정된 단계**로 만든다.  
  예: `Idle` → `AwaitingHumanMove` → `Validating` → `Committing` → `BotThinking` → …
- 상태 사이 **전이는 하나의 모듈만** 책임지게 해서, 다른 곳에서 상태를 건드리지 못하게 한다.

### 5. 봇/정책망 경로 일원화

- 봇 착수 / 정책망 폴백 / 분석 결과 선택이 서로 다른 함수에 분산되어 있다.
- **모든 착수는 하나의 합법성 경로**를 거치게 통일한다.
- 현재는 “봇이 합법성 검사로 들어오기도 하고, 안 들어오기도 하는” 불안정 상태다.

### 6. Undo/Redo와의 완전한 격리

- 착수 파이프라인 진행 중에 Undo/Redo가 들어오면 깨진다.
- 해결: Undo/Redo 요청을 **큐잉**하거나,  
  **현재 파이프라인이 끝날 때까지 블록**시키는 것.

### 7. 재현 가능한 테스트 환경

- 현재는 로그만으로 확인하고, 실제 타이밍 오류는 계속 숨어 있다.
- **“재현 시나리오”**를 자동화해서  
  **동일 입력 → 동일 결과**가 보장되는지 반복 검증해야 한다.

---

## 결론 (설계 요약)

- 대국 모드가 “한 번도 안정적으로 작동하지 않았다”는 것은  
  **구조가 레이스 컨디션을 필연적으로 만들고 있기 때문**이다.
- 작은 버그 픽스가 아니라  
  **상태 단일화 + 트랜잭션 파이프라인 + 상태 머신**으로 재설계해야 진짜 안정된다.

## 원자적 착수 파이프라인 — 구체 수정 지점 (Builder 정리)

현재 코드 기준으로 “원자적 착수 파이프라인”으로 바꾸기 위해 손봐야 할 지점을 정리한 것이다.  
**고칠 핵심 구간은 `playMove`의 낙관 업데이트와 상태 갱신 순서**이며,  
합법성 확정 후 UI 반영으로 바꾸는 것이 근본 구조 개선의 첫 단추다.

| # | 항목 | 문제·위치 | 조치 | 적용 |
|---|------|-----------|------|------|
| **1** | **낙관 업데이트 제거** | `playMove`에서 합법성 체크 전에 UI를 먼저 변경 → 레이스 유발. 위치: `EngineViewModel.playMove`. | `optimisticMoves`/`optimisticStones` 계산 및 “분석 중” 낙관 업데이트 블록을 **합법성 체크 이후로 이동하거나 삭제**. | ✅ 삭제함. 확정 전에는 game 상태 변경 없음. |
| **2** | **합법성 체크 위치** | 위치: `EngineViewModel.playMove`. | 합법성 체크를 **UI 변경 전에 항상** 수행. “합법성 OK”가 확정되기 전에는 상태를 절대 변경하지 않게 한다. | ✅ 합법성 통과 후에만 `gameUseCase.playMove` 및 `_uiState.update` 수행. |
| **3** | **moves/stones 단일화** | `moves`와 `stones`를 동시에 갱신 → 타이밍 불일치. 위치: `EngineViewModel.playMove`. | **moves만 확정 저장**, stones는 항상 moves에서 재계산(또는 캐시). 상태 갱신은 **한 번의 update 블록**에서만 수행. | ✅ 1단계에서 stones = `SimpleGoBoard.fromMoves(moves, boardSize)` 파생으로 통일. |
| **4** | **착수 트랜잭션 4단계 분리** | 위치: `EngineViewModel.playMove`. 2·3·4가 섞여 있음. | 내부를 **1. 입력 검증(좌표 범위/중복) → 2. 합법성 검증 → 3. move 확정 → 4. UI 반영** 순서로 고정. | ✅ 위 순서로 정리됨. `PlayModePhase`(VALIDATING → COMMITTING → IDLE)로 단계 구분. |
| **5** | **Undo/Redo 충돌** | 착수 중 다른 이벤트가 들어와 상태 꼬임. 위치: playMove 전체 흐름. | 착수 진행 중에는 Undo/Redo 요청 **큐잉 or 차단** (동시성 제어). | ✅ 6단계에서 `playModePhase != IDLE`이면 undo/redo/jump 계열 즉시 return. |

# 리팩터링 순서 (1단계 → 7단계)

아래 순서대로 진행하면 타이밍 레이스를 구조적으로 제거하고,  
“고치면 다른 게 깨지는” 패턴을 근본적으로 끊을 수 있다.

---

## 1) 상태 단일화 기준 확정

| 항목 | 내용 |
|------|------|
| **목표** | `moves` / `stones` 중 **하나만 진실(Source of Truth)**로 고정 |
| **제안** | `moves`를 진실로 하고, `stones`는 파생(캐시) 계산 |
| **영향 파일** | `EngineViewModel.kt`, `SimpleGoBoard.kt` |

---

## 2) 착수 트랜잭션 파이프라인 고정

| 항목 | 내용 |
|------|------|
| **목표** | 착수 흐름을 **입력 → 합법성 → 확정 → UI 반영**으로 원자화 |
| **조치** | 낙관 업데이트 제거 또는, 확정 전 UI 반영 금지 |
| **영향 파일** | `EngineViewModel.playMove` (및 관련 착수 진입점) |

---

## 3) 합법성 체크 단일 경로화

| 항목 | 내용 |
|------|------|
| **목표** | 엔진 준비 여부와 상관없이 **항상 동일한 합법성 체크 경로** 사용 |
| **조치** | 엔진 미준비 시에도 로컬 룰(SimpleGoBoard) 경로로 동일 판단 |
| **영향 파일** | `EngineViewModel.ensureMoveIsLegal` (또는 합법성 검사 담당 모듈) |

---

## 4) 대국 모드 상태 머신 도입

| 항목 | 내용 |
|------|------|
| **목표** | 상태를 명확히 분리하고, **전이 제어를 단일 모듈**로 고정 |
| **예시 상태** | `Idle` → `AwaitHuman` → `Validating` → `Committing` → `BotThinking` → `AwaitHuman` |
| **영향 파일** | `PlayViewModel.kt`, `EngineViewModel.kt` |

---

## 5) 봇/정책망 선택 경로 일원화

| 항목 | 내용 |
|------|------|
| **목표** | 봇 착수도 **항상 같은 합법성 검증 경로**를 통과 |
| **조치** | 정책망 폴백도 동일 트랜잭션 파이프라인으로 편입 |
| **영향 파일** | `PlayViewModel.selectBotMove` (및 봇/정책 착수 관련 코드) |

---

## 6) Undo/Redo 동시성 차단

| 항목 | 내용 |
|------|------|
| **목표** | 착수 트랜잭션 진행 중에는 Undo/Redo를 **큐잉 또는 차단** |
| **효과** | 충돌/되돌림 레이스 제거 |
| **영향 파일** | `EngineViewModel.playMove` (및 Undo/Redo 진입점) |

---

## 7) 재현 테스트 시나리오 고정

| 항목 | 내용 |
|------|------|
| **목표** | **동일 입력 → 동일 결과**를 보장하는 재현 케이스 확보 |
| **조치** | “자살수 / 중복 착수 / 되돌림” 3종을 자동 재현 가능하게 |
| **영향 파일** | 테스트 모듈 또는 별도 테스트 시나리오 |
