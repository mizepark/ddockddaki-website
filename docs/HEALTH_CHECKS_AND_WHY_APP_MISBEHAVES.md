# 앱이 제대로 안 돌아가는 이유 — 헬스체크·검증이 과한 부분 설명

**목적**: “지금 앱이 왜 제대로 작동을 안 하는지”, “헬스체크가 과도한 것이 있는지”를 코드 기준으로만 정리한다.

---

## 1. 헬스체크·검증이 들어가는 곳 (전부)

| 구분 | 위치 | 무엇을 함 | 실패 시 |
|------|------|-----------|----------|
| **1) init 직후 헬스체크** | KataGoEngineFacade.healthCheck | initEngine 직후 **무조건** 호출. 빈 보드로 analyzeTop4(maxVisits=2, requestId=-999, 10초 타임아웃) 한 번 돌림. | 결과가 null 또는 empty → **예외** → ensureReady/initEngineForRecovery **실패** → 엔진이 “준비 안 됨”으로 남음. |
| **2) 복구 경로 헬스체크** | initEngineForRecovery 내부 | 크래시 복구 후 init 한 세션만 하고 **동일한 healthCheck** 한 번 더. | 동일. 실패하면 복구 실패, "Engine Health Check Failed: Returned empty analysis (Zombie State)" 예외. |
| **3) JNI analyzeTop4 게이트** | katago_jni.cpp | maxVisits가 1 또는 2이고 **requestId != -999**이면 **빈 배열 반환**. (헬스체크만 requestId=-999로 허용) | 일반 분석이 실수로 1~2 visits로 호출되면 결과 없음. (현재는 maxVisits=0으로 시간 제어만 쓰므로 해당 없음) |
| **4) 매 착수 시 isLegal** | EngineViewModel.ensureMoveIsLegal | **사람이 돌을 둘 때마다** engineFacade.isLegal(세션, 수순, 좌표) 호출 → **IPC(서비스→네이티브)**. 실패 시 150ms 뒤 1회 재시도. | null/실패면 **false** → playMove에서 **착수 revert**. 엔진이 바쁘거나 응답 느리면 “착수 안 됨”처럼 보임. |
| **5) 분석 결과 게이트** | AnalysisSnapshotValidator.shouldProcessAnalysisResult | 결과마다 **4가지** 검사: (중간인데 비분석중), requestId 불일치, 보드 크기 불일치, **runEpoch < lastSubmittedEpoch**. | 하나라도 만족하면 **결과 폐기** → 추천 안 나옴. |
| **6) handleAnalysisResult 내부** | EngineViewModel | 수순 해시/길이 불일치, 타임아웃, 빈 suggestions 등으로 **return 또는 추천 비우기**. | 추천 미표시·갱신 안 됨. |

## 2. “과도하다”고 볼 수 있는 부분

### 2.1 init 직후 헬스체크 (1, 2)

- **동작**: 세션 init(분석/대국)할 때마다 **반드시** 빈 보드 분석 1번( maxVisits=2, 10초 타임아웃) 성공해야 init이 끝남.
- **과한 점**:
  - **저사양·콜드스타트**: init 직후 엔진이 10초 안에 응답 못 하면(또는 한 번이라도 빈 결과 주면) **전체 init 실패** → “엔진 준비 안 됨” 고정.
  - **복구 경로**: 크래시 복구 후에도 같은 헬스체크. 한 번 빈 결과면 "Zombie State" 예외로 **복구 자체가 실패**.
  - “엔진이 살아있는지”만 보면 되는데, **실제 분석 1번을 성공해야 통과**하는 구조라, 조건이 꽤 엄격함.
- **정리**: “헬스체크가 과도하다”에 해당. 실패 시 **앱 전체가 엔진 미준비**로 남아서, 제대로 안 돌아가는 느낌을 줌.

### 2.2 매 착수 시 isLegal (4)

- **동작**: **모든** 사람 착수에 대해 엔진에 isLegal IPC 호출. (isModelReady가 false면 호출 스킵하고 true 반환)
- **과한 점**:
  - **착수할 때마다** 엔진 mutex·서비스 왕복. 분석 중이면 직렬화 때문에 **대기**하거나 지연.
  - 엔진/서비스가 일시적으로 느리거나 실패하면 → isLegal false → **착수 revert** → “돌을 두는데 안 둬진다”는 체감.
  - 규칙상 합법성은 중요하지만, “매 수 IPC”가 **필수**인지는 정책 문제. (예: 로컬 규칙으로 먼저 검사하고, 엔진은 보조만 쓰는 식으로 완화 가능)
- **정리**: “착수 경로가 엔진 상태·응답에 과하게 의존한다”는 의미에서 과할 수 있음.

### 2.3 분석 결과 게이트 — 특히 epoch (5)

- **동작**: `updateAnalysisIntent`가 호출될 때마다 **currentEpoch++**, lastSubmittedEpoch 갱신. 들어오는 결과의 runEpoch가 lastSubmittedEpoch보다 작으면 **무조건 폐기**.
- **과한 점**:
  - 사용자 이벤트·세션 전환·재시도 등으로 **의도가 자주 갱신**되면 epoch가 계속 올라감.
  - 그 사이에 도착하는 **정상 분석 결과**가 “이전 세대”로 버려짐 → **추천이 한 번도 안 나오는** 상황 발생.
  - “이전 국면 결과는 버린다”는 목적은 맞지만, **의도 제출 빈도**가 높으면 정상 결과까지 과하게 폐기됨.
- **정리**: “epoch로 인한 폐기가 과하다”고 볼 수 있음. 앱이 제대로 안 돌아가는 원인 후보.

### 2.4 handleAnalysisResult 내부 (6)

- 수순 해시/길이 불일치, 타임아웃, 빈 suggestions 등으로 **추천을 비우거나 갱신 스킵**.
- 필요 이상으로 까다롭다기보다는, **조건이 많아서** “한 가지만 어긋나도 추천 안 나옴”이 됨. “과도한 헬스체크”라기보다 **결과 반영 조건이 많다**에 가깝고, (5)와 맞물리면 앱이 안 돌아가는 것처럼 보임.

## 3. 과도하다고 보기 어려운 부분

| 항목 | 이유 |
|------|------|
| JNI maxVisits 1~2 게이트 (3) | 잘못된 호출로 1~2 visits 결과가 나오는 경로를 막는 용도. requestId=-999 헬스체크만 통과시키고, 일반 분석은 maxVisits=0 사용이라 정상 경로에는 거의 영향 없음. |
| requestId·보드 크기 일치 (5 일부) | 다른 요청/다른 보드 결과를 붙이면 잘못된 UI. 유지하는 게 맞음. |
| 중간 결과인데 비분석중이면 폐기 (5) | 이미 분석이 끝난 뒤 도착한 중간 결과는 쓸모없음. 합리적. |

---

## 4. 요약 — “왜 제대로 안 돌아가나” / “뭐가 과한가”

- **엔진이 “준비 안 됨”으로 남는 경우**  
  → **init 직후 헬스체크(1, 2)**가 한 번이라도 실패(빈 결과/타임아웃)해서, ensureReady·복구가 실패하는 영향이 큼. **과도한 헬스체크**에 해당.

- **착수가 안 되는 것처럼 보이는 경우**  
  → **매 착수 시 isLegal(4)**이 엔진 응답 대기/실패로 false를 주면 revert됨. “매 수 IPC”가 **과하게 엄격한 전제**가 될 수 있음.

- **추천이 안 나오는 경우**  
  → **epoch 폐기(5)**로 정상 결과가 다 버려지거나, (6)의 수순/타임아웃/빈 결과 조건으로 갱신이 스킵되는 조합. **epoch 쪽이 과할 수 있음.**

- **전체적으로 “제대로 작동을 안 한다”**  
  → 위가 겹친 것: init 헬스체크 실패(엔진 미준비) + epoch/수순으로 추천 폐기 + isLegal로 착수 revert. **헬스체크·검증이 여러 겹**이라, 한 겹만 꼬여도 앱이 안 돌아가는 것처럼 보인다.
