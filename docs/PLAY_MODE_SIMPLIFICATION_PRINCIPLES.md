# 대국 모드 단순화 원칙 및 구조

“어떤 구조로 단순화해야 하는가”를 원칙과 예시로만 정리한 문서.  
구현 가이드가 아니라 **설계 방향**과 **개념 구조**를 정의한다.

---

## 단순화 원칙

| 원칙 | 설명 |
|------|------|
| **입력 경로 단일화** | 모든 탭/클릭은 **하나의 진입점**에서만 처리한다. |
| **상태 단일 진실** | 게임 상태는 `moves` 하나만 신뢰하고, 나머지는 파생한다. |
| **모드 분기 최소화** | 힌트/분석/대국 차이는 **“정책”**으로만 분리한다. (UI/입력에 분기 넣지 않음) |
| **단계적 파이프라인** | 검증 → 커밋 → 후처리 순서를 **고정**한다. |
| **비동기 의존 최소화** | 엔진은 **“결과 제공자”**로만 취급한다. (입력/검증 흐름에 엔진 상태 의존하지 않음) |

---

## 단순화 구조 예시 (개념)

### 1. InputRouter — 하나로 통합

- **역할:** “지금 모드에서 허용되는 입력인가?”만 판단한다.
- **위치:** 모든 보드 탭/버튼 입력이 **이 한 곳**을 거친다.
- **분기:** 모드(힌트/분석/대국)에 따라 “허용/무시/다른 동작”만 정책으로 반환. UI는 분기하지 않는다.

### 2. MovePipeline — 하나로 통합

- **역할:**  
  `합법성 검사 → 상태 커밋 → (필요 시) 분석 요청`  
  순서를 고정하여 한 흐름으로 처리한다.
- **분기:** “사람 착수 / 봇 착수 / 힌트 클릭” 등은 **진입 전**에 정책으로 구분되고, 파이프라인 내부는 동일 단계를 따른다.

### 3. UI — 상태 표시만 담당

- **역할:** 현재 상태를 표시한다. **입력 분기는 UI에 두지 않는다.**
- 입력 이벤트는 “어디를 눌렀는지”만 전달하고, 허용 여부와 동작 결정은 InputRouter → MovePipeline에서 한다.

---

## 기대 효과

- 서로 얽히는 로직이 줄어든다.
- 문제가 나더라도 **“어느 단계에서 실패했는지”**가 바로 드러난다.
- 오류도 줄고, 디버깅도 쉬워진다.

---

## 현재 코드와의 대응

- **원자화 + 단일 진실 + 합법성 단일 경로**  
  → `EngineViewModel.playMove` + `GameState.derivedStones()` + `ensureMoveIsLegal(SimpleGoBoard)` 로 적용됨.
- **InputRouter**  
  → `PlayViewModel.onBoardCellTapped`: 보드 탭 단일 진입점. 모드별 정책만 적용하고, `stopVariation` / `playVariation` / `playMove` 중 하나만 호출. `analysisFacade.onCellClicked` 호출 제거.
- **MovePipeline**  
  → `EngineViewModel.playMove`: 검증 → 커밋 → 후처리 고정. 주석으로 단일 파이프라인 명시.
- **UI**  
  → `PlayBoardSection`: `tapAllowed = true` 고정. 허용 여부는 InputRouter에서만 판단.

## 구조 밖 리스크(런타임 의존)

코드/구조는 안정화된 상태이고, 아래는 **구조와 별개로** 남는 실패 가능성이다.  
대국 모드가 “한 번도 제대로 작동 안 한” 것처럼 느껴진다면, 대부분 이쪽 원인일 가능성이 높다.

| 리스크 | 설명 | 영향 |
|--------|------|------|
| **엔진 준비 실패** | 엔진 초기화/준비가 끝나지 않음 | 분석·봇 응답 없음, 착수 후 처리 지연/실패 |
| **모델 로딩 실패** | 모델 다운로드·로드 실패 | 분석·봇 동작 불가 |
| **IPC 실패** | 네이티브 ↔ JVM 통신 오류 | 엔진 호출 전부 실패 가능 |
| **토큰 검증** | `playMove` 내 토큰 검사 실패 | 착수만 막힘(UI/합법성과 무관) |

- **토큰 검증**: `EngineViewModel.playMove` → `billingUseCase.ensureActiveGameTokens` 실패 시 착수 차단, `_playModePhase = IDLE` 복귀. 구조상 “어느 단계에서 막혔는지”는 파이프라인 단계로 구분 가능.
- **진단 시**: “착수 안 됨”이면 구조(InputRouter/MovePipeline)보다 **엔진 준비·모델·IPC·토큰** 순으로 확인하는 것이 효율적이다.
