# 후보 수가 적게 나오는 이유

대국모드에서 `pickGradualStrength` / `pickRapidStrength`가 “후보가 1~2개뿐”이라 best만 두는 것처럼 보일 수 있는 **원인**을 코드 경로 기준으로 정리한 문서입니다.

---

## 1. 요약 (한 줄)

**분석 시간이 짧고, MCTS가 방문을 최선 수 몇 개에만 쏟기 때문에, “방문수 > 1”인 수가 2~3개밖에 안 만들어진다. 네이티브는 그런 수만 앱으로 넘기므로 후보 수가 적게 나온다.**

---

## 2. 원인 체인 (위에서 아래로)

### 2.1 착수당 분석 시간이 짧다

- **레벨별 시간**: 프로 3~6초, 초고수6 약 2~3초, 고수1 약 1.6~2초, **최하(30) 1초** (`PLAY_MODE_BOT_SPEC.md` §3.4).
- 1초 안에는 MCTS 총 방문수가 수십~백 단위로밖에 안 쌓인다 (기기·모델에 따라 다름).

### 2.2 MCTS 특성: 방문이 소수 수에 집중된다

- MCTS(UCB/PUCT)는 **승률·정책이 좋은 수**를 자주 선택한다.
- 총 방문수가 적으면, 대부분의 방문이 **1~2개 수**에 몰리고, 나머지 수는 0~1방문만 받는다.
- 그래서 “방문수 ≥ 2”인 수가 **2~3개**만 생기는 상황이 자주 발생한다.

### 2.3 네이티브: “방문수 > 1”인 수만 앱으로 넘긴다

**위치**: `app/src/main/cpp/katago_jni.cpp` — `extractAnalysisData()`

- `search->getAnalysisData(data, 256, ...)` 로 최대 **256개** 자식(수) 정보를 가져온다.
- 그중 **`numVisits > 1`** 인 수만 float 배열에 넣어 Java로 반환한다.

```cpp
for(int i=0; i<n; i++) {
    if (data[i].numVisits <= 1) continue;  // 0 또는 1방문 수는 제외
    // ... flat[]에 기록
}
```

- 따라서 **총 방문수가 적으면** “2방문 이상”인 수가 0~3개뿐이라, **후보 개수도 0~3개**로 제한된다.
- `m == 0`이면 로그에  
  `extractAnalysisData: zero suggestions after filter n=... totalVisits=... maxVisits=...`  
  가 찍힌다.

### 2.4 Kotlin 쪽 추가 필터 (보조)

- **parseRawResult** (`AnalysisUseCase.kt`):  
  `minVisits = max(2, maxVisitsValue * minShare)` 로 한 번 더 걸러서, 이론상 후보를 더 줄일 수 있으나,  
  이미 네이티브가 “방문수 > 1”만 주기 때문에 **후보 수 감소의 주원인은 아니다**.
- **selectBotMove** (`PlayViewModel.kt`):  
  `exclude`, `visits >= BOT_MIN_SUGGESTION_VISITS(1)`, `isLegalMove` 로 일부만 제거.  
  합법 수가 1~2개만 남는 경우를 더 만들 수는 있지만, **근본 원인은 아니다**.

---

## 3. 정리 표

| 단계 | 내용 | 결과 |
|------|------|------|
| 1 | 레벨별 착수당 시간 1~6초 (최하 1초) | 총 방문수 적음 |
| 2 | MCTS가 좋은 수 위주로 탐색 | 2방문 이상인 수가 2~3개뿐 |
| 3 | 네이티브가 **numVisits > 1** 인 수만 반환 | 앱이 받는 후보 = 0~3개 수준 |
| 4 | pickGradualStrength / pickRapidStrength | 풀이 best 하나로 줄어들어 best만 두는 것처럼 보임 |

---

## 4. “후보가 많아지려면”

- **착수당 분석 시간을 늘리거나** (레벨별 maxTime 상향),
- **총 방문수가 더 쌓일 때까지 기다리거나** (봇이 “빨리 두기” 전에 조금 더 분석하게 하거나),
- **네이티브에서 “방문수 1”인 수도 넘기도록** 정책을 바꾸면 (품질/노이즈 트레이드오프 있음) 후보 수가 늘어날 수 있다.

현재 설계는 **품질을 위해 방문수 2 이상만 쓰는 것**이라, 시간이 짧을수록 후보 수가 적어지는 것이 자연스러운 결과다.
