# 현재 앱 기능 유지 + 엔진/대국모드만 simple_pad_app 스타일로 단순화 — 가능 여부

목표를 이렇게 이해했습니다.

- **유지:** UI, 분석모드·대국모드·오프라인모드·게임모드, 참고도/힌트 분석, 결제·기타 기능 **전부 그대로**.
- **바꿀 것:** 엔진이 동작하는 방식, 엔진 초기화, 대국모드의 **복잡하고 오류 나기 쉬운 구조**만 simple_pad_app을 참고해 단순화.

simple_pad_app 폴더에 둔 코드는 “이런 식으로 단순하게 쓰고 싶다”는 **참고용**이라는 전제로, **그 방향으로 단순화가 가능한지**만 설명합니다.

---

## 1. 결론: **가능하다**

- **가능합니다.**  
  UI·모드·참고도/힌트·결제 등 **기능은 전부 유지**한 채,  
  **엔진을 쓰는 방식**과 **대국/분석이 엔진을 부르는 경로**만 simple_pad_app처럼 단순하게 바꿀 수 있습니다.

이게 가능한 이유는, “단순화”를 **화면/기능 제거**가 아니라 **엔진 계층만 한 겹으로 정리**하는 것으로 보기 때문입니다.

---

## 2. 현재 “복잡하고 오류 나기 쉬운” 부분이 어디인지

코드 기준으로 보면 대략 다음과 같습니다.

- **엔진이 두 갈래**
  - `KataGoApp`: 분석용·대국용 **EngineRepository / EngineController**가 각각 따로 있음.
  - 분석 모드와 대국 모드가 **서로 다른 엔진 인스턴스(프로세스)**를 사용.
- **엔진 사용 진입점이 한 곳에 몰림**
  - `EngineViewModel`이 **3천 줄 이상**으로, 초기화·세션 전환·분석/대국 요청·콜백 등이 한 클래스에 섞여 있음.
- **UI가 엔진 디테일에 직접 의존**
  - `PlayViewModel` → `EngineViewModel` 주입.
  - `AnalysisUiViewModel`, `OfflineViewModel`, `TokenUiViewModel`도 전부 `EngineViewModel`에 의존.
  - “분석해줘”, “대국에서 한 수 둬줘”, “힌트 줘” 같은 **의도**가 아니라,  
    세션/콜백/요청 ID/어떤 Controller 쓸지 등 **내부 구현에 가까운 개념**까지 UI 쪽에서 알게 됨.
- **초기화·생명주기가 여러 곳에 흩어짐**
  - 앱에서 Controller 생성, `EngineViewModel`에서 `initEngine`·콜백 등록·shutdown/close,  
    분석/대국 전환 시 stop/start 등이 서로 얽혀 있음.

그래서 “대국모드가 복잡하고 오류 투성이”처럼 느껴지는 것은,  
**진짜 문제는 엔진 내부가 아니라 “엔진을 어떻게 쓰는가”가 일관되지 않고, 그 쓰는 경로가 한 곳(EngineViewModel)에 몰려 있기 때문**으로 보는 게 맞습니다.

---

## 3. simple_pad_app이 제시하는 “단순한 방식”

simple_pad_app에서는:

- **엔진은 인터페이스 하나**
  - `PadEngine`: `analyze(...)`, `hint(...)`, `play(...)` 같은 **suspend 함수 몇 개**만 노출.
- **요청/결과가 명확**
  - `AnalysisRequest` / `AnalysisResult`, `PlayRequest` / `PlayResult` 등 **한 세트의 모델**로만 주고받음.
- **호출하는 쪽은 “요청 → 결과”만 알면 됨**
  - 초기화·어떤 엔진 인스턴스·콜백·세션 전환 같은 건 **호출하는 쪽에서 전혀 모름**.  
  - “지금 돌일이니 분석해줘” / “대국이니 한 수 둬줘”만 넘기면, 결과만 받음.

즉, **“엔진을 쓰는 방식”만** simple_pad_app처럼 바꾸면 됩니다.  
UI·모드·참고도/힌트/결제는 그대로 두고, **그 UI가 엔진을 부를 때만** 이 단순한 방식으로 부르게 하면 됩니다.

---

## 4. “기능 유지 + 엔진/대국 구조만 단순화”가 가능한 이유

핵심은 **추상화 한 겹**을 두는 것입니다.

- **새로 둘 것**
  - **엔진 퍼사드(인터페이스)**  
    - 이름은 예: `PadEngine` 또는 `KataGoEngineFacade`.  
    - 메서드: `suspend fun analyze(...): AnalysisResult`, `suspend fun hint(...): HintResult`, `suspend fun play(...): PlayResult`  
    - (필요하면 “참고도”/“힌트”를 구분하는 건 Request 타입으로만 넣으면 됨.)
  - **구현체 하나**  
    - 예: `KataGoPadEngine` (또는 기존 이름 유지).  
    - 이 구현체 **안에서만**  
      - 지금의 `EngineController`(분석/대국 두 개),  
      - `EngineViewModel`에 있는 초기화·세션·콜백·요청 직렬화 로직  
      을 그대로 사용하거나, 점진적으로 옮겨서 정리.
- **UI/모드는 그대로**
  - `PlayViewModel`, `AnalysisUiViewModel`, 오프라인/게임 모드 등은 **기존 화면·기능 그대로** 두고,
  - **엔진이 필요할 때**  
    - 기존: `EngineViewModel`에 직접 메서드 호출, 세션/콜백/요청 ID 등 신경 씀.  
    - 변경 후: `engineFacade.analyze(request)` / `engineFacade.play(request)` 처럼 **한 번만 호출**하고, **결과만** 받음.
  - 분석모드의 “참고도/힌트 분석”, 대국모드의 “봇이 한 수”, 오프라인/게임모드의 엔진 사용도 전부 이 퍼사드를 통해서만 쓰게 하면 됨.

그래서:

- **엔진 초기화**
  - “어디서 언제 초기화하는가”는 **퍼사드 구현체 내부**로 몰면 됨.  
  - 예: `analyze`/`play`/`hint` 첫 호출 시 내부에서 `ensureEngineReady()` 한 번 호출하거나, 앱/화면 쪽에서는 “엔진 준비됐어?”만 묻는 `isReady()` 같은 것만 쓰게 할 수 있음.  
  - 즉, **초기화 방식**을 “한 곳(구현체 안)”으로 정리하는 변경이 가능합니다.
- **대국모드의 “복잡한 구성”**
  - 지금은 대국모드(PlayViewModel 등)가 `EngineViewModel`의 세션/분석/대국 전환/콜백을 직접 알고 있어서 복잡해 보이는 것이라,
  - **대국모드에서는 “한 수 달라”만 요청하고, “그 수”만 받으면 되게** 하면,  
    “복잡한 구성”은 전부 퍼사드 구현체 안(기존 EngineController·EngineViewModel 로직)으로 숨길 수 있습니다.  
  - 분석모드·참고도/힌트도 마찬가지로, 같은 퍼사드의 `analyze`/`hint`만 쓰면 됨.

따라서 **현재 앱의 모든 기능(UI, 분석/대국/오프라인/게임, 참고도·힌트, 결제 등)은 그대로 두고**,  
**엔진 작동 방식·초기화·대국모드의 복잡한 구조만** simple_pad_app을 참고해 단순화하는 것이 **가능**합니다.

---

## 5. 실제로 바꿀 때의 방향 (개념만)

- **1단계: 퍼사드 추가**
  - `PadEngine` 인터페이스(또는 동일 역할)와 Request/Result 모델을 두고,
  - 구현체 `KataGoPadEngine`에서 **기존 EngineViewModel / EngineController를 그대로 호출**하는 래퍼만 만듦.  
  - 이때 현재 API가 콜백 기반이면, 구현체 안에서 `suspendCancellableCoroutine` 등으로 “한 번 호출 → 결과 한 번 반환”으로 감싸면 됨.
- **2단계: UI는 퍼사드만 사용**
  - `PlayViewModel` 등은 `EngineViewModel` 대신 (또는 `EngineViewModel`을 “퍼사드 제공자”로만 쓰고)  
    **오직** `analyze` / `hint` / `play` 같은 메서드와 Request/Result만 보게 함.  
  - 세션/요청 ID/콜백 등은 더 이상 PlayViewModel 등에서 직접 다루지 않음.
- **3단계(선택): 구현체 안으로 로직 이전**
  - 점진적으로 EngineViewModel에 있는 초기화·세션·직렬화 로직을 `KataGoPadEngine`(또는 그 안의 전용 클래스) 쪽으로 옮기고,  
  - EngineViewModel은 “퍼사드 구현체가 쓰는 내부 컴포넌트”로만 남기거나, 점점 축소.

이렇게 하면 **기능 제거나 화면 통합 없이**, “엔진을 쓰는 방식”과 “엔진 초기화/대국 구조”만 단순화할 수 있습니다.

---

## 6. 정리

| 질문 | 답 |
|------|----|
| UI·분석/대국/오프라인/게임 모드·참고도·힌트·결제 등 **모든 기능을 유지**한 채로, 엔진 작동 방식·초기화·대국모드의 복잡한 구조만 simple_pad_app을 참고해 **단순화할 수 있는가?** | **가능하다.** |
| 어떻게? | 엔진을 **한 겹의 인터페이스(퍼사드)** 로만 쓰게 하고, 그 구현체 안에만 기존의 복잡한 로직(두 Controller, EngineViewModel, 초기화, 세션, 콜백)을 두면 된다. |
| 기능이 줄어드는가? | 아니요. 화면·모드·참고도/힌트·결제는 그대로 두고, **엔진을 부르는 경로**만 단순해진다. |

즉, simple_pad_app 폴더에 넣어둔 코드처럼 **“엔진은 analyze/hint/play 같은 단순 API로만 쓴다”**는 형태로 현재 앱의 엔진/대국 구조를 바꾸는 것은 가능하고, 그렇게 해도 현재 앱의 모든 기능은 유지할 수 있습니다.
