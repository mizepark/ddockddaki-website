# 시뮬레이션으로 발견하지 못한 이유 — 코드로 확인 가능한 문제들

## 요지

자살수·가장자리만 두기·승률 고정 같은 문제는 **엔진/파라미터가 아니라 앱 상태 동기화·폴백 로직**에서 나온 것이고, 전부 **코드만 봐도 확인 가능한** 내용이었다. 그런데도 시뮬레이션으로는 잡히지 않았다. 그 이유와, 앞으로 같은 종류의 버그를 코드 단계에서 잡는 방법을 정리한다.

---

## 1. 현재 시뮬레이션/테스트가 하는 일

### 1) Python 시뮬레이션 (`simulation_play_mode_100k.py`, `simulate_logic.py` 등)

- **실행 대상:** Kotlin/Android 코드가 아니다. **Python으로 재구현한 로직**만 돌린다.
- **검사 내용:** 레벨별 파라미터(randTemp, avoidUtility 등), 모델 선택, 시간 범위, Phase, Undo 흐름, `s_empty_suggestions`에서 "pool 비면 findAnyLegalMove" 같은 **라벨/경로 이름** 정도.
- **검사하지 않는 것:**
  - `EngineViewModel.handleAnalysisResult()`에서 해시/크기 불일치 시 **실제로 topSuggestions를 비우는지**
  - `PlayViewModel.findAnyLegalMove()`가 **(0,0)부터 순서대로** 첫 합법 수만 반환하는지 (→ 가장자리만 두는 현상)
  - `selectBotMove`가 **진짜** 합법 수만 반환하는지, 정책 배열 크기와 보드 크기가 맞는지

즉, **앱의 실제 상태 갱신·폴백 경로의 구체적 동작**은 Python 쪽에 없고, 그 부분은 시뮬레이션으로 전혀 검증되지 않았다.

### 2) Kotlin 단위 테스트 (`PlayModeSimulationTest.kt`)

- **하는 일:** Mock 엔진 + Mock 사운드로 "게임 시작 → N수 진행 → 종료" 흐름을 여러 번 돌리고, **예외 없이 끝나면 성공**으로 본다.
- **검사하지 않는 것:**
  - 실제 `PlayViewModel` / `EngineViewModel`의 `handleAnalysisResult`, `selectBotMove`, `findAnyLegalMove` 호출 여부나 결과
  - “선택된 수가 합법인지”, “topSuggestions가 수순과 일치할 때만 쓰이는지”, “폴백 시 항상 (0,0) 근처만 나오는지” 같은 **불변 조건(invariant)**

그래서 **로직이 틀려도(유령 추천, 가장자리만 두기)** “크래시만 안 나면” 시뮬레이션/테스트는 통과한다.

### 3) 로그 검증 스크립트 (`verify_bot_behavior.py`)

- **하는 일:** ADB 로그에서 봇 파라미터·착수 타입(best/analysis/fallback)·방문수 등을 파싱해 **사람이 확인**할 수 있게 해 준다.
- **하지 않는 일:**  
  “fallback만 나오면 실패”, “선택 좌표가 항상 (0,0)~(k,0)이면 실패” 같은 **자동 assertion**이 없고, **실제 앱을 돌려서 로그를 찍어봐야** 사용할 수 있다.

정리하면, **코드 레벨의 불변 조건**을 자동으로 검사하는 테스트는 없었다.

---

## 2. 왜 “코드로 확인 가능한”데도 안 잡혔는가

이번에 수정한 문제들은 다음처럼 **코드만 보면 드러나는** 성격이다.

| 문제 | 코드상 확인 가능한 내용 | 시뮬이 못 잡은 이유 |
|------|------------------------|---------------------|
| 유령 추천 (스테일 topSuggestions) | `handleAnalysisResult`에서 `analyzedMovesHash != computeMovesHash(current)` 또는 `analyzed.size != current.size`일 때 **return만 하고 topSuggestions를 비우지 않음** | Python은 해당 분기·상태를 재현하지 않음. Kotlin 테스트는 ViewModel 내부 상태를 검사하지 않음 |
| 가장자리만 두기 | `findAnyLegalMove`가 **(0,0)→(1,0)→…** 순으로 탐색해 **첫 번째** 합법 수만 반환 → 항상 같은 가장자리 | 시뮬은 “폴백 사용” 여부만 흉내 내고, **반환 좌표 순서/분포**는 검사하지 않음 |
| 승률 고정 | 위와 같이 topSuggestions가 안 비워져서 **과거 국면의 추천(승률)**만 UI에 남음 | 시뮬은 승률 값의 “현재 수순과의 일치”를 검사하지 않음 |
| 정책 배열 stride | `policy.size != boardSize*boardSize`인데도 인덱스 `y*boardSize+x`로 읽음 | Python 시뮬에 정책 배열·보드 크기 불일치 시나리오가 없음 |

즉, **“실제 앱 코드의 상태 전이와 불변 조건”**을 검사하는 테스트가 없어서, 코드만 보면 보이는 결함도 시뮬레이션 단계에서는 전혀 걸리지 않았다.

---

## 3. 같은 종류의 버그를 코드 단계에서 잡는 방법

### 3-1. 불변 조건을 명시하고 검사하는 테스트

- **EngineViewModel (또는 분석 결과 처리)**  
  - “`result.analyzedMovesHash != computeMovesHash(current)` 이면 **반드시** `topSuggestions`가 비워진 상태로 남는다.”  
  - “`analyzed.size != current.size` 이면 마찬가지로 `topSuggestions`가 비워진다.”  
  → 이걸 **단위 테스트**로 만든다:  
    Mock으로 “현재 수순 = A, 반환 결과 = B (해시/크기 불일치)”를 넣고, `handleAnalysisResult` 호출 후 `_uiState.value.analysis.topSuggestions`가 empty인지 assert.

- **PlayViewModel (폴백)**  
  - “`findAnyLegalMove`는 합법 수가 여러 개일 때 **고정된 한 칸만** 반환하면 안 된다.”  
  → 합법 수가 2개 이상인 국면을 만들어서, 여러 번 호출했을 때 **서로 다른 좌표가 나올 수 있는지** (또는 최소한 “첫 번째 합법 수”가 항상 (0,0) 근처가 아닌지) 검사.

- **정책 폴백**  
  - “`policy.size != boardSize*boardSize` 이면 정책 배열을 **사용하지 않는다** (빈 리스트 반환 등).”  
  → `loadPolicyMoves`에 `policy.size != expected`인 입력을 넣고, 빈 리스트가 나오는지 테스트.

이렇게 하면 “유령 추천”, “가장자리만 두기”, “보드 크기 불일치 시 잘못된 정책 사용”이 **코드 수정 시 회귀**해도 테스트가 실패해서 바로 드러난다.

### 3-2. 시뮬레이션에 “상태 불일치” 시나리오 추가

- Python 시뮬에서 **의도적으로** “분석 결과가 현재 수순과 다름” 상황을 넣고,  
  “이때 앱이 취해야 할 동작 = 추천 사용 금지(또는 topSuggestions 비움)”를 **명세**로 두고,  
  나중에 Kotlin 쪽에 **동일 시나리오를 재현하는 통합/단위 테스트**를 두는 방식.  
  (Python 자체는 Kotlin을 호출할 수 없으므로, “시나리오 정의 + Kotlin 테스트로 검증”이 쌍으로 가는 게 좋다.)

### 3-3. 코드 리뷰/체크리스트

- “분석 결과를 **무시하는** 모든 return 이전에, **그 결과에 묶인 UI/봇 상태(topSuggestions, isInitialAnalysisReady 등)를 정리하는가?**”
- “**폴백**으로 ‘아무 합법 수나’ 고를 때, **(0,0)부터 순서대로 첫 번째만 고르면 안 되는** 요구가 있는가? 있다면 구현이 그에 맞는가?”  
같은 질문을 PR 체크리스트에 넣어 두면, 같은 유형의 실수가 다시 들어가는 것을 줄일 수 있다.

---

## 4. 결론

- **시뮬레이션이 못 잡은 이유:**  
  - Python 시뮬은 **앱의 실제 코드를 실행하지 않고**, 파라미터·경로 이름 수준만 검사한다.  
  - Kotlin 테스트는 **예외 없이 끝나기만** 보며, **상태 불변 조건**은 검사하지 않는다.  
  - 그래서 “코드만 보면 보이는” 상태 동기화·폴백 로직 결함이 시뮬 단계에서는 전혀 걸리지 않았다.

- **앞으로 같은 종류의 문제를 코드 단계에서 잡으려면:**  
  - **불변 조건을 명시한 단위/통합 테스트** (해시·크기 불일치 시 topSuggestions 비움, findAnyLegalMove의 선택 분포, 정책 크기 검증)  
  - **“결과 무시 시 상태 정리”, “폴백 선택 방식”**을 포함한 코드 리뷰 체크리스트  
  를 도입하는 것이 좋다.

이 문서는 “지금 해결한 내용”이 아니라, **왜 시뮬로는 발견이 안 됐는지**, 그리고 **다음부터는 코드만으로도 잡을 수 있게** 하려면 무엇이 필요한지를 정리한 것이다.
