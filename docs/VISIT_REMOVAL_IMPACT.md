# 선택 항목 완전 삭제 시 영향

아래 항목들을 **완전히 삭제**했을 때의 영향을 코드 기준으로 정리함.

---

## 1. BOT_MIN_SUGGESTION_VISITS = 1 (봇 후보 방문수 1 이상만 허용)

**삭제 시:**  
- **삭제 방법**: 상수 제거 + `selectBotMove` 내 `.filter { it.visits >= BOT_MIN_SUGGESTION_VISITS }` 제거.
- **영향**:
  - 봇이 **방문수 0**인 후보도 후보 풀에 포함시켜 선택할 수 있음.
  - 엔진이 방문수 0으로 준 수(정책만 있는 수, 또는 아직 탐색 안 된 수)가 봇 착수로 나올 수 있음 → **품질 저하·노이즈/자살수 가능성 증가**.
  - 합법수 필터(`isLegalMove`)는 그대로라 불법수는 걸러지지만, **자살수·비추천 수**는 그대로 둘 수 있음.

**요약**: 봇이 “최소 1방문 이상만 고른다”는 안전장치가 사라져, 저품질/노이즈 착수 가능성이 커짐.

---

## 2. BOT_RECOVERY_MIN_VISITS = 5 (복구 시 최소 방문수 기준)

**삭제 시:**  
- **삭제 방법**: 상수 정의만 제거 (현재 **다른 코드에서 참조하지 않음**).
- **영향**:
  - **실행 중 동작 변화 없음.**  
  - 나중에 “복구 시 최소 N방문” 로직을 넣을 때 쓸 예정이었다면, 그때 5 대신 다른 상수나 설정을 써야 함.

**요약**: 사용처가 없어 **삭제해도 현재 동작에는 영향 없음.** 상수만 제거하면 됨.

---

## 3. getMaxVisits / setMaxVisits (1~3500) — 사용자 “최대 방문수” 설정

**삭제 시:**  
- **삭제 방법**: `SettingsRepository` 의 `getMaxVisits()`, `setMaxVisits(visits)` 제거 + `max_visits_dynamic` 키를 쓰는 마이그레이션/초기화 로직 제거.
- **현재 사용처**:  
  - **다른 Kotlin 파일에서 직접 호출하는 곳 없음** (검색 기준).  
  - `max_visits_dynamic` 은 마이그레이션(176~179행)에서만 읽어서 1~3500으로 보정 후 다시 저장.
- **영향**:
  - 설정 화면 등에서 “최대 방문수”를 저장/불러오는 UI가 있다면 **해당 UI가 동작하지 않거나 컴파일 에러** 발생. (현재 그런 UI가 없다면 영향 없음.)
  - 마이그레이션만 제거하면, 기존에 저장된 `max_visits_dynamic` 값은 그대로 남지만 읽는 API가 없어짐.
  - **분석 호출**은 현재 `EngineViewModel` 에서 항상 `maxVisits=0`(시간 제한만)으로 보내므로, 이 설정값이 분석 상한으로 쓰이지 않음.  
  - 단, **다른 경로**(예: 튜닝/대국/레거시)에서 `getMaxVisits()` 를 쓰도록 되어 있다면 그 부분이 깨짐.

**요약**: 지금은 **사용처가 없을 가능성이 큼**. 완전 삭제 시 해당 설정 저장/조회 기능만 사라지고, 그 설정에 의존하는 UI/로직이 있으면 그 부분만 깨짐.

---

## 4. VisitsBudgeter (목표 지연·ms/visit, 최소 16, CPU 200/500 상한)

**삭제 시:**  
- **삭제 방법**: `VisitsBudgeter` 클래스 삭제 + `EngineViewModel` 의 `private val visitsBudgeter = VisitsBudgeter(...)` 제거.
- **현재 사용처**:  
  - **`computeAdaptiveVisits` 를 호출하는 코드 없음.**  
  - `visitsBudgeter` 인스턴스만 생성되고, 메서드는 어디서도 호출되지 않음.
- **영향**:
  - **실행 중 동작 변화 없음.**  
  - “목표 지연에 맞춰 방문 수를 계산하는” 기능을 나중에 붙이려면, 같은 로직을 다시 구현하거나 다른 방식으로 대체해야 함.

**요약**: **미사용 코드**라 **삭제해도 현재 동작에는 영향 없음.** 사용하지 않는 클래스/인스턴스 제거 수준.

---

## 5. jMaxVisits 1~2 (비헬스체크) → 빈 결과 (네이티브)

**삭제 시:**  
- **삭제 방법**: `analyzeTop4` 진입부에서  
  `if (jMaxVisits >= 1 && jMaxVisits <= 2 && jRequestId != -999) { ... return env->NewFloatArray(0); }` 블록 제거.
- **영향**:
  - **분석**으로 들어온 요청이 **maxVisits=1 또는 2**로 호출되면, 지금은 빈 배열로 막는데, 삭제 후에는 **1~2방문만 돌린 결과**가 그대로 반환됨.
  - 그 결과가 UI에 “분석 결과”로 쓰이면 **추천이 거의 비거나, 1~2방문짜리만 나와서 “1로 잘린다” 현상이 다시 나타날 수 있음.**
  - 헬스체크(`requestId=-999`, maxVisits=2)는 그대로 2방문 돌리고 빈 배열이 아니어도 되므로, 이 조건만 제거해도 헬스체크 동작은 유지됨.

**요약**: **잘못된 분석 경로(1~2방문 호출)** 를 막는 방어가 사라져, 해당 경로가 있으면 “엔진 결과가 1로 잘려 나오는” 문제가 재발할 수 있음.

---

## 6. extractAnalysisData — 1짜리 필터 제거 (현재 상태 유지)

**문맥**: “1짜리 필터”는 이미 제거된 상태. 여기서 “삭제”는 **extractAnalysisData 함수 자체를 없애는 것**으로 해석.

**extractAnalysisData 함수 전체 삭제 시:**  
- **삭제 방법**: `extractAnalysisData` 함수 정의 제거 + 이 함수를 호출하는 곳 전부 제거 또는 대체.
- **호출처**:  
  - `analyzeTop4` 내부: 중간 콜백(모니터 스레드)에서 한 번, 검색 종료 후 최종 반환 시 한 번.
- **영향**:
  - **분석 결과를 Float 배열로 만드는 공통 경로가 사라짐** → 분석 결과를 Java/Kotlin으로 넘길 수 없음.
  - 중간/최종 모두 **결과를 만들 수 없어** 분석 기능이 **동작하지 않음**.  
  - “1짜리 필터만 제거”가 아니라 **함수 삭제**이면 분석 파이프라인 전체가 깨짐.

**요약**: **extractAnalysisData 는 삭제하면 안 됨.** “1짜리 필터”는 이미 제거된 상태이므로, 현재는 “필터 없이 전부 포함”만 유지하면 됨.

---

## 7. getPolicy 1/1 (정책만 1방문)

**삭제 시:**  
- **삭제 방법**: `nativeGetPolicyHeatmap`(또는 해당 JNI) 내부에서 `localParams.maxVisits = 1`, `maxPlayouts = 1` 설정 제거.  
  그러면 전역/설정 기본값(예: 1600, 1000000 등)을 쓰게 됨.
- **영향**:
  - **정책 히트맵**을 얻을 때 **1방문이 아니라 수백~수천 방문**까지 돌 수 있음.
  - 정책 요청이 **느려지고**, GPU/CPU 부하가 커짐.
  - 정책은 “빠른 착수 분포”만 필요하므로, 1/1로 두는 것이 의도된 동작.  
  - 1/1을 없애면 **정책 요청이 분석 수준으로 무거워져** 목적과 맞지 않음.

**요약**: **응답 속도·부하** 면에서만 손해. “정책만 빠르게”라는 동작이 사라지고, 정책 호출이 사실상 “짧은 분석”이 됨.

---

## 8. getRootLead·소유권 128 (리드/소유권 128 상한)

**삭제 시:**  
- **삭제 방법**: `getRootLead`, `getOwnershipHeatmap`(및 관련 네이티브) 내부에서 `localParams.maxVisits = 128`, `maxPlayouts = 128` 제거.  
  전역/설정 기본값 사용.
- **영향**:
  - 리드/소유권을 구할 때 **128이 아니라 훨씬 큰 방문 수**까지 검색할 수 있음.
  - **응답 지연 증가**, GPU/CPU 사용량 증가.
  - 리드/소유권은 “대략적인 값”만 필요하므로 128로 제한하는 것이 의도된 동작.  
  - 제한 제거 시 **리드·소유권 한 번 구할 때마다** 분석에 가까운 검색이 되어, UI 반응성·배터리 모두 나빠질 수 있음.

**요약**: **속도·리소스**만 손해. “리드/소유권만 가볍게”라는 동작이 사라짐.

---

## 표 요약

| 항목 | 삭제 시 영향 |
|------|--------------|
| BOT_MIN_SUGGESTION_VISITS=1 | 봇이 방문수 0 후보도 선택 가능 → 품질 저하·노이즈/자살수 가능성 증가 |
| BOT_RECOVERY_MIN_VISITS=5 | **현재 미사용** → 삭제해도 동작 변화 없음 |
| getMaxVisits/setMaxVisits (1~3500) | **다른 호출처 없을 수 있음** → 설정 저장/조회만 사라짐, 의존 UI 있으면 그 부분만 깨짐 |
| VisitsBudgeter | **미사용** (computeAdaptiveVisits 미호출) → 삭제해도 동작 변화 없음 |
| jMaxVisits 1~2 → 빈 결과 | 잘못된 분석 경로(1~2방문)가 다시 결과로 나옴 → “1로 잘림” 재발 가능 |
| extractAnalysisData (함수 전체) | **삭제 불가** — 분석 결과 생성 경로가 사라져 분석 기능 전체 마비 |
| getPolicy 1/1 | 정책 요청이 무거워짐 → 지연·부하 증가 |
| getRootLead·소유권 128 | 리드/소유권 요청이 무거워짐 → 지연·부하 증가 |

---

이 문서는 “해당 항목을 완전히 삭제했을 때”의 영향만 기술함. 부분 수정(예: 상수 변경)은 다루지 않음.
